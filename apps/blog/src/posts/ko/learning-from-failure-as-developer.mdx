# 실패에서 배우는 개발자

NestJS 프로젝트를 Docker로 배포하다가 온갖 에러와 씨름하던 밤이었습니다. 처음에는 "대충 하면 되겠지"라는 마음으로 시작했는데, 나중에는 무슨 수를 써도 안 될 것 같은 순간이 찾아왔습니다.

그런데 결국 해냈습니다. 그리고 그 과정에서 Docker 네트워킹, 멀티스테이지 빌드, 환경변수 관리에 대해 문서 100개를 읽는 것보다 더 많이 배웠습니다.

## 공자의 세 가지 깨달음

공자는 깨달음의 단계를 세 가지로 나누었습니다:

1. **태어나면서 아는 사람** - 천재
2. **배워서 아는 사람** - 학습자  
3. **겪어서 아는 사람** - 실천가

어릴 때는 자만심이 높아서 스스로를 최소한 "배워서 아는 사람" 정도는 된다고 생각했습니다. 이제는 "겪어서라도 알 수 있게 된다면" 감사하려 하고 있습니다.

## 요즘 개발자들의 함정

요즘에는 뭘 해도 전부 알고서 시작하려는 사람이 많습니다:

> "먼저 Docker 완벽 가이드 다 보고 시작해야지"  
> "TypeScript 고급 문법 마스터하고 프로젝트 시작할게"  
> "일단 클린 아키텍처 책 한 번 더 읽고..."

하지만 전부 안다고 경지에 도달할 수 있는 건 아닙니다. 오히려 뭣도 모르고 시작한 게 꾸준하게 시간을 들인다면 장인의 경지에 오르게 됩니다.

## 적당한 실패의 가치

적당한 실패에서 오는 **투지와 분노**는 많은 정보보다 경지에 오르는 데 도움이 됩니다.

```typescript
// 처음: "이게 왜 안 돼?"
try {
  await prisma.user.create({ data: userData });
} catch (e) {
  console.log(e); // P2002: Unique constraint failed
  // 아... unique 제약조건이 뭐지?
}

// 나중: 실패를 통해 배운 패턴
try {
  await prisma.user.upsert({
    where: { email: userData.email },
    create: userData,
    update: userData,
  });
} catch (e) {
  if (e instanceof Prisma.PrismaClientKnownRequestError) {
    if (e.code === 'P2002') {
      throw new ConflictException('이미 존재하는 이메일입니다');
    }
  }
  throw e;
}
```

처음에는 "왜 안 되지?" 하며 에러 메시지를 구글링했습니다. 수십 번의 실패 후에는 에러 코드만 봐도 원인을 알게 됩니다. 이건 문서를 읽어서 얻은 지식이 아니라, **몸으로 배운 지식**입니다.

## 실패하는 방법도 기술이다

모든 실패가 가치 있는 건 아닙니다. 의미 있는 실패를 위한 조건:

### 1. 결과를 만들어내려는 태도

```bash
# 나쁜 실패: 에러 나면 포기
$ docker build .
ERROR: failed to solve...
$ # "아 몰라, 나중에 하지 뭐"

# 좋은 실패: 끝까지 추적
$ docker build .
ERROR: failed to solve...
$ docker build . --no-cache --progress=plain 2>&1 | tee build.log
$ # 로그 분석, 레이어별 확인, 원인 파악
```

### 2. 실패 후 기록

```markdown
## 2024-12-20 Docker 배포 실패 기록

### 증상
- `COPY --from=builder` 단계에서 파일을 찾지 못함

### 원인
- 멀티스테이지 빌드에서 `npm run build`가 `dist/` 대신 `build/`에 출력
- tsconfig.json의 `outDir` 설정 확인 안 함

### 해결
- `outDir: "./dist"` 명시적 설정
- Dockerfile의 COPY 경로와 일치시킴

### 배운 것
- 암묵적 설정에 의존하지 말 것
- 빌드 결과물 경로는 항상 명시적으로 확인
```

### 3. 패턴 인식

세 번 이상 같은 유형의 실패를 겪으면, 그건 더 이상 실패가 아니라 **학습 기회를 낭비하는 것**입니다.

## 개발자 5년차의 깨달음

개발자로 5년을 일하며 느낀 것:

**1년차**: "난 뭘 모르는지도 몰라"  
**2년차**: "아, 내가 많이 모르는구나"  
**3년차**: "이제 좀 알 것 같아"  
**4년차**: "아직도 모르는 게 더 많네"  
**5년차**: "모르는 게 당연하고, 배우면 된다"

요즘 주니어 개발자들을 보면 "처음부터 완벽하게 해야 해"라는 압박감에 시달리는 것 같습니다. 하지만 시니어들도 매일 모르는 것을 마주합니다. 차이점은 **모르는 것을 마주했을 때의 태도**입니다.

## 실천 가이드

### 오늘 당장 할 수 있는 것

1. **TODO 앱이라도 만들기 시작하세요**
   - "완벽한 아키텍처"를 기다리지 마세요
   - 일단 만들고, 부족함을 느끼고, 개선하세요

2. **실패 로그를 기록하세요**
   - 블로그, 노션, 어디든 좋습니다
   - 같은 실패를 반복하지 않는 게 목표입니다

3. **"왜?"를 세 번 물어보세요**
   - 에러가 났다 → 왜?
   - 타입이 안 맞았다 → 왜?
   - API 응답 스키마를 확인 안 했다 → 왜?

## 마무리

> 실패에서 무엇을 배울 수 있을까? 
> 이건 직접 실패해보기 전까지는 절대 알 수 없는 영역입니다.

그럼에도 결과를 만들어 내려는 태도와 행동이 "배움"이라는 걸 줍니다. 

완벽한 준비란 없습니다. 시작하세요. 실패하세요. 기록하세요. 그게 성장입니다.
