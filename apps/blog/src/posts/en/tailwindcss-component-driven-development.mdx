# Component-Driven Development with TailwindCSS

Component-driven development has become essential in modern frontend development. **TailwindCSS**, with its utility-first approach, makes component-based publishing even more efficient. I'll share TailwindCSS utilization techniques and component structuring strategies from real-world experience.

## 🎯 Why TailwindCSS + Component Approach?

### Limitations of Traditional CSS Approaches

- **CSS File Bloat**: Becomes harder to manage as projects grow
- **Naming Conflicts**: Even methodologies like BEM and OOCSS aren't perfect
- **Unused CSS**: Accumulation of styles that aren't actually used

### Advantages of TailwindCSS

- **Utility-First**: Use only the styles you need by composing utilities
- **Consistent Design System**: Predefined spacing, colors, and typography
- **Tree-shaking**: Unused classes are automatically removed
- **Responsive-First**: Mobile-first approach feels natural

### 2. Real Implementation Example: Button Component

```typescript
// components/atoms/Button/Button.tsx
import { cn } from "@/lib/utils";
import { VariantProps, cva } from "class-variance-authority";
import { ButtonHTMLAttributes, forwardRef } from "react";

const buttonVariants = cva(
  // Base styles
  "inline-flex items-center justify-center rounded-md text-sm font-medium transition-colors focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-offset-2 disabled:pointer-events-none disabled:opacity-50",
  {
    variants: {
      variant: {
        default: "bg-blue-600 text-white hover:bg-blue-700",
        secondary: "bg-gray-100 text-gray-900 hover:bg-gray-200",
        outline: "border border-gray-300 bg-white hover:bg-gray-50",
        ghost: "hover:bg-gray-100",
      },
      size: {
        sm: "h-8 px-3 text-xs",
        md: "h-10 px-4",
        lg: "h-12 px-8 text-base",
      },
    },
    defaultVariants: {
      variant: "default",
      size: "md",
    },
  }
);

interface ButtonProps
  extends ButtonHTMLAttributes<HTMLButtonElement>,
    VariantProps<typeof buttonVariants> {}

const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ className, variant, size, ...props }, ref) => {
    return (
      <button
        className={cn(buttonVariants({ variant, size, className }))}
        ref={ref}
        {...props}
      />
    );
  }
);

export { Button, buttonVariants };
```

### 3. Responsive Card Component

```typescript
// components/molecules/Card/Card.tsx
interface CardProps {
  title: string;
  description: string;
  image?: string;
  className?: string;
  children?: React.ReactNode;
}

export const Card = ({
  title,
  description,
  image,
  className,
  children,
}: CardProps) => {
  return (
    <div
      className={cn(
        // Base card styles
        "bg-white rounded-lg shadow-sm border border-gray-200",
        // Hover effects
        "hover:shadow-md hover:border-gray-300 transition-all duration-200",
        // Responsive padding
        "p-4 sm:p-6",
        className
      )}
    >
      {image && (
        <div className="aspect-video w-full mb-4 overflow-hidden rounded-md">
          <img src={image} alt={title} className="w-full h-full object-cover" />
        </div>
      )}

      <h3 className="font-semibold text-lg sm:text-xl text-gray-900 mb-2">
        {title}
      </h3>

      <p className="text-gray-600 text-sm sm:text-base mb-4 line-clamp-3">
        {description}
      </p>

      {children}
    </div>
  );
};
```

## 📱 Responsive Web Publishing Practical Tips

### 1. Mobile-First Approach

TailwindCSS is mobile-first by default. Define styles for small screens first and progressively enhance for larger screens.

```html
<!-- Mobile: vertical layout, Tablet+: horizontal layout -->
<div class="flex flex-col md:flex-row gap-4">
  <div class="w-full md:w-1/2">Left content</div>
  <div class="w-full md:w-1/2">Right content</div>
</div>

<!-- Responsive grid -->
<div
  class="grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-6"
>
  <!-- Cards -->
</div>
```

## 🔧 Frequently Used Patterns in Production

### 1. Compound Component Pattern

```typescript
// Flexible card composition with compound component pattern
const Card = {
  Root: ({ children, className }: CardProps) => (
    <div className={cn('bg-white rounded-lg shadow-sm border', className)}>
      {children}
    </div>
  ),

  Header: ({ children }: { children: React.ReactNode }) => (
    <div className="px-6 py-4 border-b border-gray-200">
      {children}
    </div>
  ),

  Content: ({ children }: { children: React.ReactNode }) => (
    <div className="px-6 py-4">
      {children}
    </div>
  ),

  Footer: ({ children }: { children: React.ReactNode }) => (
    <div className="px-6 py-4 border-t border-gray-200 bg-gray-50">
      {children}
    </div>
  ),
}

// Usage
<Card.Root>
  <Card.Header>
    <h3>Title</h3>
  </Card.Header>
  <Card.Content>
    <p>Content</p>
  </Card.Content>
  <Card.Footer>
    <Button>Action</Button>
  </Card.Footer>
</Card.Root>
```

### 2. Conditional Styling

```typescript
// Utility function for conditional styling
const getButtonStyles = (variant: string, isLoading: boolean) => {
  return cn("px-4 py-2 rounded-md font-medium transition-colors", {
    "bg-blue-600 hover:bg-blue-700 text-white": variant === "primary",
    "bg-gray-200 hover:bg-gray-300 text-gray-800": variant === "secondary",
    "opacity-50 cursor-not-allowed": isLoading,
    "hover:opacity-80": !isLoading,
  });
};
```

## 📝 Conclusion

TailwindCSS and component-driven development are at the core of modern frontend development. They provide significant advantages in terms of **reusability**, **consistency**, and **maintainability**.

### Key Insights from Production Experience:

1. **Start Small**: Build systematically from Atom-level components
2. **Design System First**: Establish consistent tokens and style guides
3. **Consider Performance**: Remove unnecessary styles aggressively
4. **Don't Forget Accessibility**: Use semantic HTML and appropriate ARIA attributes
5. **Establish Team Conventions**: Unify class name ordering, naming rules, etc.

When explaining these practical experiences with specific examples in interviews, you'll definitely make a good impression!

Particularly approaching with the perspective that **"it's not just a CSS replacement, but a tool that improves the overall development experience together with component-based design philosophy"** would be great.
